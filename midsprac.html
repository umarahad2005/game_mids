🔁 Update Loops
WHEN DOES MY CODE RUN?

Unity has different loop methods. Understanding which one to use and why is the key exam point.

MethodWhen it runsUse it for
Start() Once, before first frame Initializing variables, getting component references
Update() Every frame (variable rate) Input detection, movement, non-physics logic
FixedUpdate() Fixed timestep (50x/sec default) Rigidbody physics, forces, velocity changes
LateUpdate() After all Update() calls Camera follow (so it moves after player has moved)
Awake() Before Start(), even if disabled Singleton setup, initializing references between scripts
// EXAM SCENARIO 1

"Your player character stutters when moving on different computers." — This happens because you move in Update() without Time.deltaTime. The fix is either multiply movement by deltaTime, or move physics in FixedUpdate().

// EXAM SCENARIO 2

"Your camera jitters when following the player." — Move camera logic to LateUpdate() so it runs after the player has already moved in Update().

⏱️ Time.deltaTime
FRAME-RATE INDEPENDENT MOVEMENT

The Problem: Without deltaTime, your game runs faster on a 120fps machine than a 30fps machine. A player moving at "5 units/frame" moves 600 units/sec on 120fps but only 150 units/sec on 30fps.

What deltaTime is: The time in seconds since the last frame. On 60fps ≈ 0.016s. On 30fps ≈ 0.033s.

The Math: speed × Time.deltaTime = units per second (regardless of frame rate)

// BAD — speed depends on frame rate void Update() { transform.Translate(Vector3.forward * speed); } // GOOD — speed is always same per second void Update() { transform.Translate(Vector3.forward * speed * Time.deltaTime); }
// WHY IT MATTERS IN FPS GAMES

In your FPS game, if bullet speed or player movement doesn't use deltaTime, the game becomes unplayable at different frame rates. Always multiply movement values by Time.deltaTime when inside Update().

In short questions: if a scenario says the game behaves differently on fast vs slow machines — the answer is Time.deltaTime. If it says physics is involved — the answer is FixedUpdate().

💥 Triggers & Collisions
THE MOST SCENARIO-HEAVY TOPIC

Both detect when GameObjects touch, but they work differently and are used in different situations.

CollisionTrigger
Objects stop each other?✅ Yes (physical)❌ No (pass through)
Requires Rigidbody?On one objectOn one object
Collider settingNormal collider"Is Trigger" ✅ checked
Methods usedOnCollisionEnter/Stay/ExitOnTriggerEnter/Stay/Exit
Game usesWalls, floors, physics objectsPickups, checkpoints, damage zones
// COLLISION — physical contact, objects bounce void OnCollisionEnter(Collision collision) { if (collision.gameObject.CompareTag("Enemy")) { health -= 10; } } // TRIGGER — ghost zone, objects pass through void OnTriggerEnter(Collider other) { if (other.gameObject.CompareTag("Pickup")) { ammo += 10; Destroy(other.gameObject); } }
// EXAM SCENARIO 1

"You want a door to open when the player walks near it, but the player shouldn't be blocked by an invisible wall." — Use a Trigger. The player passes through the trigger zone, OnTriggerEnter fires, door opens.

// EXAM SCENARIO 2

"A bullet hits an enemy and the enemy should take damage." — For the actual physical hit, use OnCollisionEnter. But in FPS games, bullets are often very fast so you might use Raycast instead (no collider needed).

// EXAM SCENARIO 3

"You have a coin pickup in your FPS game. The player should collect it by touching it, and the coin should disappear." — Trigger. Check Is Trigger on coin's collider. Use OnTriggerEnter on coin or player. Call Destroy(gameObject).

Key rule: If the objects should physically push/stop each other → Collision. If one should pass through the other (detection only) → Trigger. Always remember: at least one object in a Trigger/Collision pair needs a Rigidbody.
♾️ Singleton Pattern
ONE INSTANCE TO RULE THEM ALL

A Singleton ensures only one instance of a script exists in the entire game, and any other script can access it globally.

// WHY USE SINGLETON?

Imagine a GameManager — it tracks score, lives, and game state. You don't want two GameManagers accidentally running at the same time (that would double-count score, etc.). Singleton guarantees exactly one exists and every script can reach it easily.

public class GameManager : MonoBehaviour { // Static instance — shared across entire game public static GameManager Instance; public int score = 0; void Awake() { // If no instance exists, this becomes the instance if (Instance == null) { Instance = this; DontDestroyOnLoad(gameObject); // persist across scenes } // If one already exists, destroy this duplicate else { Destroy(gameObject); } } } // Any other script accesses it like this: GameManager.Instance.score += 10;
// EXAM SCENARIO 1

"You are building an FPS game. The score, ammo count, and player health need to be tracked and displayed, accessible from any script in the game." — Use a Singleton GameManager. One instance persists throughout the game, any script can call GameManager.Instance.score etc.

// EXAM SCENARIO 2

"When the player picks up ammo, the ammo counter script needs to update, and the gun script also needs to know about the change." — A Singleton bridges both. Both scripts call GameManager.Instance.ammo.

Awake() is used (not Start()) for Singleton setup because Awake runs before any Start(). This ensures the instance exists before other scripts try to access it in their Start().
The 3 key things about Singleton: (1) static Instance variable, (2) check in Awake() if instance already exists, (3) DontDestroyOnLoad if you want it to survive scene changes.
💾 PlayerPrefs
SAVING DATA BETWEEN GAME SESSIONS

PlayerPrefs saves data to the device (like a tiny key-value database). It survives closing and reopening the game.

Data TypeSaveLoadDefault if not found
intPlayerPrefs.SetInt("key", value)PlayerPrefs.GetInt("key")0
floatPlayerPrefs.SetFloat("key", value)PlayerPrefs.GetFloat("key")0f
stringPlayerPrefs.SetString("key", value)PlayerPrefs.GetString("key")""
// SAVING high score void SaveHighScore() { PlayerPrefs.SetInt("HighScore", score); PlayerPrefs.Save(); // optional, auto-saves on quit anyway } // LOADING high score when game starts void Start() { int savedScore = PlayerPrefs.GetInt("HighScore", 0); // 0 = default } // DELETE saved data PlayerPrefs.DeleteKey("HighScore"); PlayerPrefs.DeleteAll(); // wipe everything
// EXAM SCENARIO 1

"In your FPS game, the player's best score should still be shown when they reopen the game." — Use PlayerPrefs.SetInt("HighScore", score) to save, and PlayerPrefs.GetInt("HighScore", 0) to load. This persists across sessions.

// EXAM SCENARIO 2

"The player customized their sensitivity setting. It should be remembered next time." — PlayerPrefs.SetFloat("Sensitivity", value) on change, GetFloat on Start().

PlayerPrefs is NOT for large amounts of data or secure data (like passwords). It's only for small, simple settings and scores. For complex save systems, you'd use JSON/files, but that's beyond this exam.
PlayerPrefs key difference from Singleton: Singleton lives during ONE game session in memory. PlayerPrefs lives on the DEVICE PERMANENTLY until deleted. Singleton dies when you close the game. PlayerPrefs doesn't.

🏃 Rigidbody Movement (FPS)
HOW THE PLAYER ACTUALLY MOVES

In an FPS game, the player uses a Rigidbody for physics-based movement. You read input, then apply velocity or force to the Rigidbody.

// WHY RIGIDBODY AND NOT TRANSFORM.TRANSLATE?

Transform.Translate teleports the object — it ignores walls and physics. Rigidbody movement respects colliders, gravity, and friction. In an FPS game the player must be blocked by walls and affected by gravity, so Rigidbody is required.

public class PlayerMovement : MonoBehaviour { [SerializeField] private float speed = 5f; [SerializeField] private float jumpForce = 5f; private Rigidbody rb; void Start() { rb = GetComponent<Rigidbody>(); // get reference once } // FixedUpdate for all Rigidbody movement void FixedUpdate() { float h = Input.GetAxis("Horizontal"); // A/D keys float v = Input.GetAxis("Vertical"); // W/S keys Vector3 move = new Vector3(h, 0f, v) * speed; rb.velocity = new Vector3(move.x, rb.velocity.y, move.z); } void Update() { // Input detection in Update (not FixedUpdate) if (Input.GetKeyDown(KeyCode.Space)) { rb.AddForce(Vector3.up * jumpForce, ForceMode.Impulse); } } }
// EXAM SCENARIO

"Why is the player movement code inside FixedUpdate() instead of Update() in your FPS game?" — Because we're modifying rb.velocity (Rigidbody). Physics calculations must be in FixedUpdate() which runs at a fixed 50Hz timestep, synchronized with the physics engine. Putting physics in Update() can cause inconsistent, jittery results.

Important distinction: Input.GetKeyDown() should be in Update() (so you don't miss short button presses). Physics (rb.velocity, rb.AddForce) should be in FixedUpdate(). You can use a flag/bool to communicate between them.
🔫 Shooting — Instantiate & Destroy
SPAWNING AND REMOVING GAME OBJECTS

Instantiate creates a copy of a prefab at runtime. Destroy removes a GameObject from the scene.

// WHY PREFABS?

A prefab is a saved template of a GameObject (like a bullet). You design one bullet in the editor, save it as a prefab, then Instantiate spawns as many copies as needed while the game runs.

public class Gun : MonoBehaviour { [SerializeField] private GameObject bulletPrefab; // drag prefab here [SerializeField] private Transform firePoint; // where bullet spawns [SerializeField] private float bulletSpeed = 20f; void Update() { if (Input.GetButtonDown("Fire1")) { // left mouse click Shoot(); } } void Shoot() { // Create bullet at firePoint position and rotation GameObject bullet = Instantiate( bulletPrefab, firePoint.position, firePoint.rotation ); // Push bullet forward Rigidbody rb = bullet.GetComponent<Rigidbody>(); rb.velocity = firePoint.forward * bulletSpeed; // Auto-destroy after 3 seconds (clean up) Destroy(bullet, 3f); } }
// EXAM SCENARIO 1

"Why do we Destroy bullets after a few seconds even if they haven't hit anything?" — Because every active GameObject takes memory and processing. Bullets flying forever off-map cause a memory leak and performance drop. Destroy(bullet, 3f) auto-cleans after 3 seconds.

// EXAM SCENARIO 2

"You need the bullet to disappear when it hits an enemy." — Add a Bullet script with OnCollisionEnter. When tag is "Enemy", deal damage and call Destroy(gameObject).

// On the Bullet script void OnCollisionEnter(Collision col) { if (col.gameObject.CompareTag("Enemy")) { col.gameObject.GetComponent<EnemyHealth>().TakeDamage(25); } Destroy(gameObject); // destroy bullet regardless of what was hit }
👁️ Mouse Look / Camera Rotation
LOOKING AROUND IN FPS
public class MouseLook : MonoBehaviour { [SerializeField] private float mouseSensitivity = 100f; [SerializeField] private Transform playerBody; // player root private float xRotation = 0f; void Start() { // Lock and hide cursor Cursor.lockState = CursorLockMode.Locked; } void Update() { float mouseX = Input.GetAxis("Mouse X") * mouseSensitivity * Time.deltaTime; float mouseY = Input.GetAxis("Mouse Y") * mouseSensitivity * Time.deltaTime; xRotation -= mouseY; xRotation = Mathf.Clamp(xRotation, -90f, 90f); // limit up/down transform.localRotation = Quaternion.Euler(xRotation, 0f, 0f); playerBody.Rotate(Vector3.up * mouseX); // left/right turns body } }
// WHY TWO SEPARATE ROTATIONS?

In an FPS, left/right (Y axis) rotates the entire player body — so movement also changes direction. Up/down (X axis) only rotates the camera, not the body — so you look up without the player tilting. These are separate transforms: camera script is on the Camera, body rotation is on the Player GameObject.

📄 Complete Movement Script
using UnityEngine; public class PlayerMovement : MonoBehaviour { // SerializeField = editable in Inspector, protected from other scripts [SerializeField] private float moveSpeed = 5f; [SerializeField] private float jumpForce = 5f; [SerializeField] private Transform groundCheck; [SerializeField] private LayerMask groundLayer; private Rigidbody rb; private bool isGrounded; private bool jumpRequest = false; void Start() { // Get reference to Rigidbody component on same object rb = GetComponent<Rigidbody>(); } void Update() { // Input detection MUST be in Update to catch every key press if (Input.GetKeyDown(KeyCode.Space) && isGrounded) { jumpRequest = true; // flag for FixedUpdate } // Ground check — sphere cast downward isGrounded = Physics.CheckSphere( groundCheck.position, 0.2f, groundLayer ); } void FixedUpdate() { // Physics/Rigidbody in FixedUpdate for consistency float h = Input.GetAxis("Horizontal"); float v = Input.GetAxis("Vertical"); Vector3 direction = transform.right * h + transform.forward * v; rb.velocity = new Vector3( direction.x * moveSpeed, rb.velocity.y, // preserve Y (gravity) direction.z * moveSpeed ); if (jumpRequest) { rb.AddForce(Vector3.up * jumpForce, ForceMode.Impulse); jumpRequest = false; } } }
📄 Complete Gun / Shooting Script
using UnityEngine; public class Gun : MonoBehaviour { [SerializeField] private GameObject bulletPrefab; [SerializeField] private Transform firePoint; [SerializeField] private float bulletSpeed = 30f; [SerializeField] private float bulletLifetime = 3f; [SerializeField] private int maxAmmo = 30; private int currentAmmo; void Start() { currentAmmo = maxAmmo; } void Update() { if (Input.GetButtonDown("Fire1") && currentAmmo > 0) { Shoot(); } } void Shoot() { // Instantiate: creates a copy of prefab in the scene GameObject bullet = Instantiate(bulletPrefab, firePoint.position, firePoint.rotation); // Apply velocity to push bullet forward Rigidbody rb = bullet.GetComponent<Rigidbody>(); if (rb != null) rb.velocity = firePoint.forward * bulletSpeed; // Schedule destruction — prevents memory leaks Destroy(bullet, bulletLifetime); currentAmmo--; } }
📄 Singleton GameManager
using UnityEngine; public class GameManager : MonoBehaviour { // Static = belongs to class, not any instance public static GameManager Instance; public int score = 0; public int ammo = 30; public int health = 100; void Awake() { // Awake runs before Start — critical for singleton if (Instance == null) { Instance = this; DontDestroyOnLoad(gameObject); // survive scene loads } else { Destroy(gameObject); // kill duplicate } } public void AddScore(int points) { score += points; PlayerPrefs.SetInt("HighScore", Mathf.Max(score, PlayerPrefs.GetInt("HighScore", 0))); } }
// How any other script uses it: GameManager.Instance.AddScore(10); GameManager.Instance.health -= 20;
📄 Bullet Script (with Collision + Destroy)
using UnityEngine; public class Bullet : MonoBehaviour { [SerializeField] private int damage = 25; // Called when this bullet physically hits something void OnCollisionEnter(Collision collision) { // Check what we hit using tags if (collision.gameObject.CompareTag("Enemy")) { EnemyHealth enemy = collision.gameObject.GetComponent<EnemyHealth>(); if (enemy != null) enemy.TakeDamage(damage); } // Destroy bullet on ANY collision (wall, floor, enemy) Destroy(gameObject); } } // Separate EnemyHealth script public class EnemyHealth : MonoBehaviour { public int health = 100; public void TakeDamage(int amount) { health -= amount; if (health <= 0) Die(); } void Die() { GameManager.Instance.AddScore(100); // Singleton access! Destroy(gameObject); } }

Which Unity method runs at a fixed timestep and is best for Rigidbody physics? OBJECTIVE ▶
// ANSWER FixedUpdate() — it runs at a consistent 50 times per second (by default), synchronized with Unity's physics engine. This ensures physics calculations don't vary with frame rate.
What does [SerializeField] do that the "public" keyword does NOT? OBJECTIVE ▶
// ANSWER [SerializeField] shows a private variable in the Inspector but keeps it inaccessible from other scripts. Public also shows in Inspector but lets any script read and change it — less safe/encapsulated.
What is the key difference between OnTriggerEnter and OnCollisionEnter? OBJECTIVE ▶
// ANSWER OnCollisionEnter — objects physically block each other. OnTriggerEnter — one object has "Is Trigger" checked, so they pass through each other; only detection happens, no physical response.
PlayerPrefs.GetInt("Score") returns what if the key has never been saved? OBJECTIVE ▶
// ANSWER Returns 0 (the default value). You can also specify a custom default: PlayerPrefs.GetInt("Score", 50) would return 50 if the key doesn't exist.
── SHORT QUESTIONS (SCENARIO-BASED) ──
SCENARIO: You are making an FPS game. The player moves faster on a high-end PC with 120fps than on a laptop with 30fps. How do you fix this? SHORT ▶
// ANSWER This is a frame-rate dependency issue. The fix is to multiply all movement values by Time.deltaTime.

Time.deltaTime is the time in seconds since the last frame. By multiplying speed × Time.deltaTime, you convert "units per frame" to "units per second," which is the same on all machines regardless of frame rate.

rb.velocity = direction * speed; → BAD
transform.Translate(direction * speed * Time.deltaTime); → GOOD

Note: If using Rigidbody in FixedUpdate(), the physics timestep is already fixed, so deltaTime is less critical there — but it's still good practice.
SCENARIO: You want a coin in your FPS level to disappear and increase the score when the player walks over it. The player should NOT be blocked by the coin. What do you use and why? SHORT ▶
// ANSWER Use a Trigger (not Collision) because the player should pass through the coin without being physically blocked.

Steps: (1) On the coin's Collider, check "Is Trigger". (2) On the coin script, use OnTriggerEnter. (3) When the player tag is detected, add to score via GameManager.Instance, then call Destroy(gameObject) on the coin.

Collision would have been wrong here because it would block the player's movement.
SCENARIO: Your FPS game needs a score system. Multiple scripts (gun, enemy, pickup) all need to update and read the score. How would you architect this? SHORT ▶
// ANSWER Use the Singleton pattern with a GameManager class.

A Singleton ensures only one GameManager exists in the scene. It has a public static GameManager Instance that any script can access without needing a reference.

Any script just calls GameManager.Instance.score += 10. Without Singleton, each script would need a reference dragged in the Inspector, which is messy and breaks easily.
SCENARIO: The player's high score should still be visible when they reopen the game tomorrow. How do you implement this? SHORT ▶
// ANSWER Use PlayerPrefs. Unlike variables which die when the game closes, PlayerPrefs saves data to the device persistently.

Save: PlayerPrefs.SetInt("HighScore", score);
Load on Start: int hs = PlayerPrefs.GetInt("HighScore", 0);

The second argument in GetInt is the default value if the key doesn't exist yet (first time playing).
SCENARIO: Bullets in your FPS game slow down the game after 30 seconds of play. What is the likely cause and fix? SHORT ▶
// ANSWER Memory leak / too many active GameObjects. Bullets are being Instantiated but never Destroyed. After many shots, hundreds of bullets are flying off-screen, each using CPU and memory.

Fix: Destroy(bullet, 3f); in the Shoot() method. This auto-destroys the bullet 3 seconds after spawning if it hasn't hit anything yet.

Also: Destroy the bullet in OnCollisionEnter when it hits something.
SCENARIO: You want to move the player using Rigidbody. Should you put the movement code in Update() or FixedUpdate()? Explain why. SHORT ▶
// ANSWER FixedUpdate() for Rigidbody movement. Physics in Unity runs on a fixed timestep (default 0.02s, 50 times/sec). If you modify rb.velocity or AddForce in Update() which runs at variable frame rates, the physics calculations become inconsistent and jittery.

However, Input detection (GetKeyDown, GetButtonDown) should still be in Update() because FixedUpdate might skip frames and miss short button presses. Use a bool flag to pass the input from Update() to FixedUpdate().

// ANSWER — see the Scripts tab for the complete script Key points to mention in your answer:

Write a Gun script that fires bullets using Instantiate. Explain how Instantiate and Destroy are used and why each is needed. LONG ▶
// ANSWER — see the Scripts tab for full script Key points to cover:
  • Instantiate(prefab, position, rotation) — creates a runtime copy of the bullet prefab at the firePoint. Without prefabs, you'd have to pre-place every bullet in the scene which is impossible
  • GetComponent<Rigidbody>() on the spawned bullet — to apply velocity (propel it forward)
  • rb.velocity = firePoint.forward * bulletSpeed — direction comes from the firepoint transform, which is aimed forward
  • Destroy(bullet, 3f) — if bullet hits nothing, it gets cleaned up after 3 seconds. Without this, bullets accumulate and cause performance drops (memory leak)
  • Ammo check: if(currentAmmo > 0) prevents firing with no ammo
Write a GameManager script using the Singleton pattern. Explain why Awake() is used and what DontDestroyOnLoad does. LONG ▶
// ANSWER — see Scripts tab for full script Key explanation points:
  • public static GameManager Instance — static means it belongs to the class itself, not any specific object. So GameManager.Instance works from anywhere without needing a reference
  • Awake() not Start() — Awake runs before Start(), so the Instance is ready before any other script's Start() tries to access it. If you used Start() for Singleton setup, other Start() methods might run first and find Instance is null
  • if(Instance == null) — first time: set Instance to this. Second time (duplicate): Destroy it. This guarantees exactly one GameManager
  • DontDestroyOnLoad(gameObject) — when a new scene loads, Unity normally destroys all objects. This line tells Unity to keep the GameManager alive across scene changes
